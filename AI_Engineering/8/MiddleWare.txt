// Middleware in Express: reusable functions that process requests/responses
// They run before your route handlers and can modify req/res or control flow.

// 1. app.use(cors());
//
// Definition: Enables Cross-Origin Resource Sharing (CORS).
// Why we need it: Allows your frontend (different domain/port) to talk to your backend.
// Analogy: Security guard at a building gate with a guest list.
//   → By default, only locals are allowed. CORS lets visitors from other neighborhoods in.

// 2. app.use(express.json());
//
// Definition: Parses incoming requests with JSON payloads.
// Why we need it: Without it, req.body is undefined when client sends JSON.
// Analogy: Translator who converts client’s JSON (foreign language) into plain English.
//   → Makes sure your server understands the client’s message.

// 3. app.use(express.static(__dirname));
//
// Definition: Serves static files (HTML, CSS, JS, images) from the given directory.
// Why we need it: Lets you host frontend files directly without writing extra routes.
// Analogy: Librarian handing out books directly from the shelf.
//   → Visitors can pick up index.html, style.css, or images without asking the librarian to read them aloud.

// Middleware Concept (general):
//
// Definition: Functions that sit in the middle of request → response pipeline.
// Why we need it: To add features, run code on every request, or control flow.
// Analogy: 
//   - Security checkpoints at an airport (passport, baggage, customs).
//   - Assembly line stations adding parts to a car.
//   - Filters in a coffee machine processing liquid before it reaches your cup.
//
// Flow: Request → Middleware stack → Route handler → Response

        Middleware (simple explanation)
        - When you run an Express server, a client (like your browser or a frontend app) sends a request.
        - The server eventually sends back a response.
        - Middleware is everything that happens in the middle — little helpers that check, change, or add things before the response goes out.
