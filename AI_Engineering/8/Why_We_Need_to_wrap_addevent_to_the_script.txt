### Causes we identified
1. **Event listener not attaching**
   - If your script runs before the DOM is ready, `document.getElementById("gift-form")` returns `null`.
   - Then `giftForm.addEventListener("submit", ‚Ä¶)` fails, so `preventDefault()` never runs.
   - Result: the form submits normally ‚Üí `/?`.

2. **`marked` import breaking the script**
   - If the browser can‚Äôt resolve `import { marked } from "marked";` (because `node_modules` isn‚Äôt exposed or bundled), the script throws an error.
   - That stops execution before the event listener attaches.
   - Again, the form submits normally ‚Üí `/?`.

3. **Static serving setup**
   - `app.use(express.static(__dirname))` already serves `index.html`, JS, and CSS.
   - Adding `app.get("/")` is redundant unless you want custom logic.
   - Exposing `node_modules` with `app.use("/node_modules", ‚Ä¶)` is not required and insecure in production.

### Solutions we discussed
- **Ensure DOM readiness**
  - Wrap code in `DOMContentLoaded`, OR
  - Load your script with `defer` in `<head>`, OR
  - Place your `<script>` at the bottom of `<body>`.
  - All three guarantee the DOM exists before your script runs.

- **Fix `marked` import**
  - Use a bundler (Vite/Webpack/Parcel) so `marked` is bundled into your frontend JS.
  - Or import from a CDN:
    ```js
    import { marked } from "https://cdn.jsdelivr.net/npm/marked/marked.min.js";
    ```
  - This avoids exposing `node_modules`.

- **Simplify Express setup**
  - Keep:
    ```js
    app.use(express.static(__dirname));
    ```
  - Drop the redundant `app.get("/")` unless you need custom behavior.
  - Don‚Äôt expose `node_modules` unless you have a very specific reason.

### Key takeaway
You saw `/?` because your script failed before attaching the form handler.  
That failure was caused either by running too early (DOM not ready) or by the `marked` import breaking.  
Ensuring DOM readiness and fixing the `marked` import resolves the issue.
