# ----------------------------
# 1. What is an environment variable?
# ----------------------------
# Environment variables are key-value pairs stored by the operating system.
# They act like hidden settings that programs can read at runtime.
# Example: PATH tells Windows where to look for executables.

# Note:
# Never access env variables from frontend like script.js ie Always store env var in Backend like server.js

# ----------------------------
# 2. Why and when to use them?
# ----------------------------
# - To configure applications without hardcoding values in code.
# - To store secrets (like API keys or DB passwords) securely outside source files.
# - To make apps portable across dev, test, and production environments.
# - To avoid editing code when switching configurations.

# ----------------------------
# 3. Why are they considered safer?
# ----------------------------
# - Secrets stay OUT of source code and version control (GitHub, etc.).
# - Prevents accidental leaks when sharing code with others.
# - They can be injected at runtime (CI/CD pipelines, Docker, Kubernetes).
# - Developers can use different credentials locally without changing code.
# ⚠️ Note: They are safer than hardcoding, but not bulletproof.
#   -> Use secret managers (Azure Key Vault, AWS Secrets Manager, HashiCorp Vault)
#      for production-grade protection.

# 4. Setting environment variables in PowerShell (temporary, session-only)
# ----------------------------
$env:DB_HOST = "localhost"   # Database host
$env:DB_USER = "myuser"      # Database username
$env:DB_PASS = "mypassword"  # Database password

Detailed Theory -

		In software development, there are **two primary types of environments**: **development** (the local machine for coding and testing) and **production** (the live server where the app runs for end users).

		### **Why Separate Environments?**
		*   **Safety:** Test changes without breaking the live application.
		*   **Performance:** Optimize each setup for speed or debugging.
		*   **Configuration:** Apply different settings (like database links) to each environment.

		### **Code Implementation: `process.env`**
		You access environment variables in Node.js via the `process.env` object.

		```javascript
		// Set a port with a fallback value
		const port = process.env.PORT || 3000;

		// Access specific variables
		const dbUrl = process.env.DATABASE_URL;
		const nodeEnv = process.env.NODE_ENV;

		// Verify settings
		console.log(`Running on port ${port}`);
		console.log(`Environment: ${nodeEnv}`);
		```

		### **Environment Configurations (`.env`)**

		| Feature            | **Development Setup**          | **Production Setup**          |
		| **`NODE_ENV`**     | `development`                  | `production`                  |
		| **`PORT`**         | `3000`                         | `80`                          |
		| **`DATABASE_URL`** | Local MongoDB instance         | Cloud-based database          |
		| **`DEBUG`**        | `true`                         | `false`                       |
		| **`LOG_LEVEL`**    | `verbose`                      | `error`                       |
		| **Focus**          | Errors, logging, hot reloading | Performance, minimal exposure |

		### **Managing Variables with `dotenv`**
		The `dotenv` package loads variables from a file into your app.

		**Install:**
		```bash
		npm install dotenv
		```

		**Initialize (Top of app):**
		```javascript
		require('dotenv').config();
		```

		### **Security Best Practices**
		*   **`.gitignore`:** Always add `.env` here so it is **never committed** to a repository.
		*   **`.env.example`:** Create a template file to show required variables to other developers.
		*   **Vault Services:** Use these to secure secrets in production.
		*   **Documentation:** Use clear naming and descriptions for all variables.
