## 🧩 What is Zod?
- **Definition**: Zod is a **TypeScript-first schema validation library**.  
- **Purpose**: It lets you define the *shape* of your data (what fields exist, what types they are) and then automatically validate inputs against that shape.  
- **Why we use it**: In MCP servers, tools need to accept inputs from an AI model. You don’t want garbage or wrong types slipping in. Zod ensures the inputs are exactly what you expect.  

### Example:  
```ts
import { z } from "zod";

// Define a schema for a city input
const citySchema = z.string().describe("City name (New York or London)");

// Validate input
citySchema.parse("London"); // ✅ works
citySchema.parse(123);      // ❌ throws error, not a string
```

So Zod acts like a **gatekeeper**: only valid inputs pass through.

## 🏗️ MCP Server Basics
Think of an MCP server as a **toolbox** that an AI can call.  
- **Server instance**: The toolbox itself (`new MCPServer(...)`).  
- **Tools**: Individual tools inside the box (like “get weather data”).  
- **Resources & Prompts**: Extra features we’ll add later.  

---

## ⚙️ Step-by-Step: Weather Tool
Here’s the **full annotated code** for your weather fetcher tool:

```ts
// server.ts
import { MCPServer } from "@modelcontextprotocol/sdk"; 
import { z } from "zod";

// 1. Create the server instance
const server = new MCPServer({
  name: "weather-data-fetcher",
  version: "1.0.0",
});

// 2. Helper function to simulate weather data
async function getWeatherByCity(city: string) {
  if (city.toLowerCase() === "new york") {
    return { city, temperature: 22, forecast: "Partly cloudy with a breeze" };
  } else if (city.toLowerCase() === "london") {
    return { city, temperature: 18, forecast: "Light rain showers" };
  } else {
    throw new Error(`Weather data not available for ${city}`);
  }
}

// 3. Register the tool
server.tool(
  {
    name: "getWeatherDataByCityName",
    description: "Fetch mock weather data for New York or London",
    inputSchema: z.object({
      city: z.string().describe("City name (New York or London)"),
    }),
  },
  async ({ city }) => {
    try {
      const weatherData = await getWeatherByCity(city);
      return {
        content: [
          {
            type: "text",
            text: JSON.stringify(weatherData),
          },
        ],
      };
    } catch (err: any) {
      return {
        content: [
          {
            type: "text",
            text: `Error: ${err.message}`,
          },
        ],
      };
    }
  }
);
```

---

## 🔍 Breaking Down the Code
- **`MCPServer`**: Creates the server (like starting a toolbox).  
- **`server.tool(...)`**: Registers a tool inside the server.  
- **`z.object({ city: z.string() })`**: Defines the input schema → tool expects a string called `city`.  
- **Handler function**: Runs when the tool is called. It fetches weather data and returns JSON text.  
- **Error handling**: If the city isn’t New York or London, it returns an error message.  

---

## 🧠 Why This Matters
- Without Zod, if the AI sends `city: 123`, your tool would break.  
- With Zod, you catch bad inputs early and return a clear error.  
- This makes your MCP server **robust and predictable**.
