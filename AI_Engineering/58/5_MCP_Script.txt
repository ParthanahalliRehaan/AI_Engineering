## 🌐 What is MCP?
**MCP = Model Context Protocol**  
It’s an **open standard** that defines how applications can provide **context** (extra information) to large language models (LLMs) like GPT, Claude, or Gemini.

- **Model** → the LLM itself.  
- **Context** → the extra info you give the model (e.g., your name, recent data, files).  
- **Protocol** → the rules for how this info flows between apps and models.

---

## ⚡ Why MCP is Needed
LLMs have two big limitations:
1. **No recent knowledge** → They don’t know what happened today because training data is old.  
2. **Context is expensive** → Feeding huge amounts of data (like all today’s news) costs a lot in tokens.

👉 MCP solves this by letting models access **just the right data, at the right time, in a standardized way**.

---

## 🔌 Analogy: USB-C for AI
Think of MCP like **USB-C**:
- USB-C lets you plug in chargers, headphones, laptops with one standard port.  
- MCP lets you plug in **tools and data sources** (weather API, Google search, local PDFs, calendar) into your LLM app in a standard way.

---

## 🏗️ How MCP Works
MCP uses a **client-server architecture**:

- **MCP Client** → lives inside your host app (like an IDE or chat app).  
- **MCP Server(s)** → connect to external data sources (weather API, local files, Google search, private APIs).  
- Each server knows how to **fetch data**, package it as **context**, and send it back to the model.  

You can have multiple servers:
- Server A → Weather data  
- Server B → Local files  
- Server C → Google search  

All of them feed context into the client, which then helps the LLM answer with **fresh, relevant info**.

---

## 🧩 Recap
- **MCP = Model Context Protocol**  
- **Purpose** → Give LLMs access to live tools & data.  
- **Problems solved** → Lack of recent knowledge + expensive context windows.  
- **How** → Client-server setup where servers fetch and package context.  
- **Benefit** → Efficient, modular, plug-and-play access to external data.
