## The "can I go to the site and do everything as postgres?" question

**No** — and this is really important to understand.

The `postgres` role never touches your frontend or your JavaScript code directly. Here's the mental model:

```
Your JS code  →  uses API key  →  Supabase decides which role to apply
```

Supabase has **two API keys**:

- `anon` key — safe to use in frontend, applies the `anon` role
- `service_role` key — **secret**, bypasses ALL security, applies `postgres`-level access

So when you write `supabase.from("embeddings").insert(...)` in your JS, Supabase is checking **which key you initialized the client with** — that determines the role, not you manually "logging in" as postgres.

The Supabase **dashboard** is where you act as postgres — creating tables, running SQL, managing everything. That's your superuser playground. Your app code should never have that power.

---

## Adding RLS to your embeddings table

Since your project currently has no user auth (just inserting embeddings), we'll set up a simple but realistic policy. Here's the plan:

- Enable RLS on the table (this **locks everything down** by default)
- Allow `anon` role to insert (so your script can still add rows)
- Allow `anon` role to select (so you can read them back)

Run this in your Supabase **SQL Editor**:

```sql
-- Step 1: Enable RLS on the table
alter table embeddings enable row level security;

-- Step 2: Allow anyone (anon) to INSERT new embeddings
create policy "Allow anon insert"
on embeddings
for insert
to anon
with check (true);

-- Step 3: Allow anyone (anon) to SELECT/read embeddings
create policy "Allow anon select"
on embeddings
for select
to anon
using (true);
```

---

## What `using` and `with check` mean

These two are RLS-specific and trip up a lot of beginners:

**`using (true)`** — applies to **reading** rows (SELECT). It's the filter that decides *which rows* a role can see. `true` means "all rows are visible."

**`with check (true)`** — applies to **writing** rows (INSERT/UPDATE). It validates *the data being written*. `true` means "any data is allowed."

Later, when you add auth, you'd write something like:

```sql
using (auth.uid() = user_id)
-- "only show rows where the logged-in user's ID matches"
```

But for now, `true` is the right call since you have no users yet.

---

## After adding RLS — test your insert script again

Your JS code doesn't need to change at all. Supabase will now check the policy before allowing the insert, and since you allowed `anon` inserts, it'll go through just like before — but now it's **properly secured** with a real policy in place rather than no security at all.
