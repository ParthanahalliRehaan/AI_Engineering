## You vs. Your Users

| | You (Creator) | Regular Users |
|---|---|---|
| **Where** | Supabase Dashboard | Your website |
| **Role** | `postgres` | `anon` or `authenticated` |
| **Can do** | Everything | Only what policies allow |
| **How** | Dashboard / SQL Editor | Your JS code + API key |

---

## How it flows when you add Auth

Right now your table has no concept of "whose row is this." When you add auth, the flow becomes:

```
User signs up/logs in
       ↓
Supabase gives them a session (JWT token)
       ↓
Their requests now carry their unique user ID (auth.uid())
       ↓
RLS policies check that ID before allowing anything
```

So you'd add a `user_id` column to your table and update your policies like:

```sql
-- Only see YOUR OWN rows
create policy "Users see own rows"
on embeddings
for select
to authenticated
using (auth.uid() = user_id);

-- Only insert rows that belong to YOU
create policy "Users insert own rows"
on embeddings
for insert
to authenticated
with check (auth.uid() = user_id);
```

Now even if two users are in the same table, **they are completely blind to each other's data** — the database enforces it, not your JS code.

---

## The key insight

Your JS code stays almost the same. You're not manually passing the user ID around or writing `WHERE user_id = ...` in every query. Supabase automatically reads the logged-in user's token and applies the RLS policy behind the scenes.

That's the beauty of RLS — **security lives in the database, not scattered across your app code.**
