## ⏳ 1. Image URLs and Expiration
- When you generate an image with DALL·E, by default you get a **URL** pointing to the image.  
- That URL **expires after 1 hour**.  
- If you don’t save the image in that time, the link breaks and you’ll only see the alt text.  

👉 **Best practice:** download the image immediately if you want to keep it.

---

## 🌐 2. Using the URL Format
This is the simplest way to display generated images in a webpage.

```js
document.body.innerHTML = `
  <img src="${image.data[0].url}" 
       alt="curious hedgehog exploring a garden">
`;
```

- `image.data[0].url` → temporary URL to the image.  
- `alt` → descriptive text (same as your prompt).  

⚠️ Limitation: The image disappears after 1 hour.

---

## 🔒 3. Permanent Access with Base64 JSON
To avoid expiration, you can request images in **Base64 JSON format**.

### What is Base64?
- Base64 is a way of converting binary data (like an image file) into a long string of ASCII characters.  
- Example snippet: `iVBORw0KGgoAAAANSUhEUgAA...`  
- Browsers can decode this string back into the original image.

### Why `data:image/png;base64,...`?
This prefix is **essential**:
- `data:` → tells the browser the source is inline data, not a URL.  
- `image/png` → specifies the file type (PNG image).  
- `base64` → tells the browser the data is Base64 encoded.  
- After the comma comes the actual Base64 string.  

So the full `src` looks like:

```html
<img src="data:image/png;base64,iVBORw0KGgoAAAANSUhEUgAA..." 
     alt="tiny curious hedgehog with quills shimmering in the sunlight exploring a lush garden">
```

Without this prefix, the browser wouldn’t know how to interpret the string.

---

## ⚖️ 4. URL vs Base64 — Trade‑offs
| Format | How it Works | Pros | Cons | Best Use |
|--------|--------------|------|------|----------|
| **URL** | Temporary link to hosted image | Lightweight, easy to share | Expires in 1 hour | Quick previews, social sharing |
| **Base64 JSON** | Encoded image data embedded directly | Permanent, no hosting needed | Huge strings, can crash small browsers | Embedding in apps, educational material |

---

## 🛠️ 5. Example Code — Both Methods
Here’s a **ready‑to‑use snippet** showing both approaches:

```js
// Using URL (expires in 1 hour)
document.body.innerHTML = `
  <img src="${image.data[0].url}" 
       alt="curious hedgehog exploring a garden">
`;

// Using Base64 (permanent)
document.body.innerHTML = `
  <img src="data:image/png;base64,${image.data[0].b64_json}" 
       alt="curious hedgehog exploring a garden">
`;
```

---

## ⚠️ 6. Important Notes
- Base64 strings are **massive** — embedding large images can slow down or crash lightweight browsers (like Scrimba’s mini browser).  
- Works fine on your local machine or in production apps.  
- To reduce risk, generate **smaller images** (e.g., `256x256`) when using Base64.  

---

## 📌 7. Summary
1. **URL format** → quick, temporary (expires in 1 hour).  
2. **Base64 JSON format** → permanent, embedded directly, but heavy.  
3. `data:image/png;base64,...` → required prefix so browsers know how to decode and render the Base64 string.  
4. Use **URL for short‑term sharing**, **Base64 for long‑term embedding**.
