## 1. Important acronyms (practical list)

You do NOT need to memorize everything now; just recognize them.

- DB / DBMS – Database / Database Management System (software that stores and manages data).  
- RDBMS – Relational DBMS (data in tables with rows & columns, e.g., PostgreSQL). [aws.amazon](https://aws.amazon.com/what-is/object-relational-mapping/)
- SQL – Structured Query Language, language to talk to relational DBs. [aws.amazon](https://aws.amazon.com/what-is/object-relational-mapping/)
- CRUD – Create, Read, Update, Delete, the four basic operations on data. [crowdstrike](https://www.crowdstrike.com/en-us/cybersecurity-101/observability/crud/)
- ACID – Atomicity, Consistency, Isolation, Durability (rules that keep data correct and safe). [scribd](https://www.scribd.com/presentation/58265136/Database-Management-System)
- ORM – Object‑Relational Mapping / Mapper (library that maps JS objects ↔ database tables). [prisma](https://www.prisma.io/dataguide/types/relational/what-is-an-orm)
- API – Application Programming Interface, how frontend ↔ backend talk (usually over HTTP).  
- JWT – JSON Web Token, a signed token used for authentication (Supabase Auth uses JWT). [youtube](https://www.youtube.com/watch?v=9T6tjTQ4Zo4)
- RLS – Row Level Security, Postgres feature to control which rows each user can see. Supabase uses this heavily. [supabase](https://supabase.com/docs/guides/database/overview)

Supabase/Postgres specific:

- Postgres / PostgreSQL – The relational database engine Supabase is built on. [supabase](https://supabase.com/docs/guides/database/overview)
- Schema – Namespace/group for tables (e.g., `public` schema). [supabase](https://supabase.com/blog/natural-db)

## 2. New words and simple meanings

These are words you’ll see all the time in Supabase/Postgres docs.

- Table – Like an Excel sheet: rows = records, columns = fields. [supabase](https://supabase.com/docs/guides/database/overview)
- Row / Record – One entry in a table (one user, one task, one order). [aws.amazon](https://aws.amazon.com/what-is/object-relational-mapping/)
- Column / Field – One property (name, email, created_at). [supabase](https://supabase.com/docs/guides/database/overview)
- Primary Key (PK) – Unique ID for each row (usually `id`). [aws.amazon](https://aws.amazon.com/what-is/object-relational-mapping/)
- Foreign Key (FK) – A column that references another table’s primary key, for relationships. [aws.amazon](https://aws.amazon.com/what-is/object-relational-mapping/)
- Relationship – How tables connect (one‑to‑many, many‑to‑many, etc.). [aws.amazon](https://aws.amazon.com/what-is/object-relational-mapping/)
- Index – Extra structure on a column to speed up queries. [supabase](https://supabase.com/docs/guides/resources)
- Constraint – Rule on data, e.g., `NOT NULL`, `UNIQUE`, foreign key rules. [scribd](https://www.scribd.com/presentation/58265136/Database-Management-System)
- Query – A request to the DB (usually an SQL statement or equivalent via Supabase client). [supabase](https://supabase.com/docs/guides/database/overview)
- Transaction – Group of queries that succeed or fail together (money transfer example). [dbvis](https://www.dbvis.com/thetable/sql-transactions-in-a-flask-crud-application/)
- Migration – Versioned change to the schema (add table, change column, etc.).  
- View / Function / Trigger – Extra SQL objects for computed data or automatic logic. [supabase](https://supabase.com/docs/guides/resources)

Supabase side:

- Supabase Client – JS library (`@supabase/supabase-js`) you use in frontend or backend to talk to Postgres. [youtube](https://www.youtube.com/watch?v=9T6tjTQ4Zo4)
- Row Level Security (RLS) Policy – SQL rule on a table that decides which rows a user can access. [supabase](https://supabase.com/solutions/beginners)

## 3. What does a table do in backend? (beginner → advanced)

### Beginner: basic job of a table

- Store data as rows.  
- Support CRUD operations: [devart](https://www.devart.com/dbforge/sql/sqlcomplete/crud-operations-in-sql.html)
  - Create: add new rows (`INSERT`).  
  - Read: get rows (`SELECT`).  
  - Update: modify existing rows (`UPDATE`).  
  - Delete: remove rows (`DELETE`).

In Supabase, same idea but via JS:

```js
// Create
const { data, error } = await supabase
  .from('tasks')
  .insert({ title: 'Learn Supabase' })

// Read
const { data, error } = await supabase
  .from('tasks')
  .select('*')
```

### Intermediate: structure & relationships

- Define types: `text`, `integer`, `boolean`, `timestamp`, etc.  
- Add primary keys so each row is uniquely identified. [aws.amazon](https://aws.amazon.com/what-is/object-relational-mapping/)
- Connect tables with foreign keys, e.g., `tasks.user_id` references `users.id`. [aws.amazon](https://aws.amazon.com/what-is/object-relational-mapping/)
- Use indexes on columns used often in `WHERE` clauses to speed up searches. [supabase](https://supabase.com/docs/guides/resources)
- Add defaults (`created_at` default `now()`). [supabase](https://supabase.com/docs/guides/resources)

### Advanced: rules and performance

- Constraints to enforce data rules (e.g., `email` UNIQUE). [scribd](https://www.scribd.com/presentation/58265136/Database-Management-System)
- Row Level Security (RLS) so each logged‑in user only sees their own data. [supabase](https://supabase.com/solutions/beginners)
- Triggers/functions to auto‑run logic when rows are inserted/updated. [supabase](https://supabase.com/docs/guides/resources)
- Transactions for grouped operations that must not partially fail. [dbvis](https://www.dbvis.com/thetable/sql-transactions-in-a-flask-crud-application/)

You don’t need the advanced part on day 1; just know it exists.

## 4. What is an ORM (and related concepts)?

### Plain‑English idea

ORM = “translator” between your **objects in code** and **tables in DB**.  
Instead of writing raw SQL strings, you mostly work with JS objects and methods. [geeksforgeeks](https://www.geeksforgeeks.org/dbms/what-is-object-relational-mapping-orm-in-dbms/)

Without ORM:

```js
// Node + pg driver
const result = await pool.query(
  'SELECT * FROM users WHERE id = $1',
  [userId]
)
const user = result.rows[0]
```

With ORM:

```js
// Example using Prisma-style API
const user = await prisma.user.findUnique({
  where: { id: userId }
})
```

The ORM:

- Knows that `User` class ↔ `users` table. [geeksforgeeks](https://www.geeksforgeeks.org/dbms/what-is-object-relational-mapping-orm-in-dbms/)
- Knows fields like `email`, `password`, etc.  
- Generates SQL behind the scenes for you. [theserverside](https://www.theserverside.com/definition/object-relational-mapping-ORM)

Benefits:  

- Work with JS objects instead of raw SQL. [freecodecamp](https://www.freecodecamp.org/news/what-is-an-orm-the-meaning-of-object-relational-mapping-database-tools/)
- Less repetitive code for joins/relations. [theserverside](https://www.theserverside.com/definition/object-relational-mapping-ORM)
- Often better type safety and migrations.

Important: **Supabase is not exactly an ORM**, but its JS client feels ORM‑like:

```js
const { data, error } = await supabase
  .from('users')
  .select('*')
  .eq('id', userId)
```

So when you hear “ORM/DB layer/DB client” in practice, think:  
> “The code that lets my backend talk to the database in a convenient way.”

## 5. Where do we connect Supabase: backend or frontend?

Supabase is flexible:

### Option A: Frontend + Supabase only

- You use Supabase JS client directly in React/Next. [youtube](https://www.youtube.com/watch?v=dU7GwCOgvNY)
- Database security is handled by RLS policies and auth. [supabase](https://supabase.com/solutions/beginners)
- Good for simple apps, prototypes, dashboards.

Example (React):

```js
import { createClient } from '@supabase/supabase-js'

const supabase = createClient(
  import.meta.env.VITE_SUPABASE_URL,
  import.meta.env.VITE_SUPABASE_ANON_KEY
)

async function loadTasks(userId) {
  const { data, error } = await supabase
    .from('tasks')
    .select('*')
    .eq('user_id', userId)
}
```

### Option B: Backend (Node/Express) + Supabase + frontend

- Frontend talks to your backend (Express routes).  
- Backend uses Supabase client (with service role key) to talk to Postgres. [github](https://github.com/orgs/supabase/discussions/2125)
- Use this when you:  
  - Need secret business logic (payments, admin actions).  
  - Need to call other services (Stripe, email, etc.) plus DB.  
  - Want to hide powerful DB keys from browser.

Example (Node/Express backend):

```js
// supabaseClient.js (server-side only)
import { createClient } from '@supabase/supabase-js'

export const supabase = createClient(
  process.env.SUPABASE_URL,
  process.env.SUPABASE_SERVICE_ROLE_KEY // keep on server
)
```

```js
// routes/users.js (Express)
import { supabase } from './supabaseClient.js'

app.get('/api/users/:id', async (req, res) => {
  const { id } = req.params

  const { data, error } = await supabase
    .from('users')
    .select('*')
    .eq('id', id)
    .single()

  if (error) return res.status(400).json({ error: error.message })
  res.json(data)
})
```

Frontend then just calls `/api/users/123` instead of touching Supabase directly.

## 6. How to integrate “tables + ORM/DB client” into backend logic

General flow for a typical backend route:

1. Frontend sends HTTP request to your backend (`/api/todos`, `/api/orders`).  
2. Express route handler receives the request (`req`).  
3. Inside that handler, you use **DB layer** (Supabase client or ORM) to:  
   - Create / read / update / delete rows from tables (CRUD). [en.wikipedia](https://en.wikipedia.org/wiki/Create,_read,_update_and_delete)
4. You send back JSON response with data or error.  

Pseudo‑code:

```js
app.post('/api/tasks', async (req, res) => {
  const { title, userId } = req.body

  // TALK TO DB (Supabase/ORM)
  const { data, error } = await supabase
    .from('tasks')
    .insert({ title, user_id: userId })

  if (error) return res.status(400).json({ error: error.message })
  res.json(data)
})
```

The **table** is just where data lives.  
The **ORM / Supabase client** is how your backend code talks to that table.  
The **route handlers** are your “backend logic.”
